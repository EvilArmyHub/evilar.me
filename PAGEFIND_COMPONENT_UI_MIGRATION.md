# Pagefind Component UI Migration Plan

## Why This Is Worth Doing

The current search works, but it is maintained through custom modal code around
`@pagefind/default-ui`. Migrating to Pagefind Component UI is useful because the
new package provides maintained web components for modal/searchbox behavior,
shared search state, keyboard behavior, WAI-ARIA-oriented UI patterns, and
automatic text localization based on the page language.

This is not a bug fix and does not need to be bundled with unrelated upstream
work. Treat it as a separate search refactor.

Relevant docs:

- <https://pagefind.app/docs/search-ui/>
- <https://pagefind.app/docs/components/>
- <https://pagefind.app/docs/components/config/>

## Current State

- The index is generated by `pagefind --site dist` in `postbuild`.
- Runtime UI uses `@pagefind/default-ui`.
- `src/components/Search.astro` renders a custom wrapper, button, and dialog.
- `src/scripts/site-search.ts` owns modal open/close behavior, `Ctrl/Cmd+K`,
  focus management, and lazy initialization.
- `src/styles/blocks/search.css` targets `.pagefind-ui__*` selectors and
  `--pagefind-ui-*` variables.
- The current search UI is localized and integrated with the custom header, so
  migration must preserve the Russian UX and compact header behavior.

## Target State

- Replace `@pagefind/default-ui` with `@pagefind/component-ui`.
- Keep the Pagefind CLI and `postbuild` indexing workflow.
- Use Pagefind web components in `Search.astro`, including:
  - `<pagefind-config>`
  - `<pagefind-modal-trigger>`
  - `<pagefind-modal>`
- Remove custom modal JavaScript only after the built-in component behavior is
  verified against the existing UX.
- Rewrite search CSS from `.pagefind-ui__*` / `--pagefind-ui-*` to the Component
  UI classes and `--pf-*` variables.
- Keep search scoped to existing `data-pagefind-body` content and existing
  `data-pagefind-filter` tag filters.

## Best-Practice Decisions

- Do the migration in a dedicated change set. Search UI has visible behavior,
  keyboard behavior, accessibility concerns, and CSS surface area.
- Use `pnpm` only, and update `pagefind` and `@pagefind/component-ui` together
  so generated assets and imported components stay in the same release family.
- Do not merge or copy upstream wholesale. Use upstream and official docs as
  references, then adapt to this site's component split and design tokens.
- Prefer the maintained Component UI modal over custom modal plumbing if it
  preserves the current UX. Reducing custom JavaScript is the main value of this
  migration.
- Keep Russian language behavior as an acceptance criterion, not an assumption.

## Implementation Steps

1. Dependencies:
   - Add `@pagefind/component-ui`.
   - Remove `@pagefind/default-ui`.
   - Update `pagefind` to the matching current stable version.
   - Remove the local `@pagefind/default-ui` module declaration if no longer used.

2. Components:
   - Replace `Search.astro` internals with Component UI elements.
   - Preserve the outer spacing and header alignment contract.
   - Remove `SearchButton.astro`, `SearchDialog.astro`, and
     `src/scripts/site-search.ts` after parity is confirmed.

3. Styling:
   - Replace `@pagefind/default-ui/css/ui.css` with
     `@pagefind/component-ui/css`.
   - Port colors, focus states, result link styling, backdrop blur, and compact
     trigger sizing to Component UI selectors and variables.

4. Behavior:
   - Confirm modal open/close behavior.
   - Confirm keyboard shortcut behavior or intentionally document any difference.
   - Confirm focus lands in the search input when the modal opens.
   - Confirm clicking a result closes or navigates cleanly.
   - Confirm production-only search behavior remains understandable in local dev.

## Acceptance Criteria

- `pnpm check` passes.
- `pnpm build` passes and `postbuild` creates the Pagefind index.
- Search opens from the header on desktop and mobile.
- Search UI text is Russian or otherwise acceptable for `lang="ru-RU"`.
- Results still include posts and notes only.
- Tag filters still work through `data-pagefind-filter`.
- No stale references to `@pagefind/default-ui`, `.pagefind-ui__*`, or
  `site-search.ts` remain after migration.
